From a task sentence to an edit plan with codixing ask

An agent rarely starts with a symbol name. It starts with a sentence: “release: verify assets and website.” Here is how Codixing v0.47.1 turns that sentence into ranked evidence, likely tests, explicit risks, and a bounded path to the next tool call.

Start with the task you actually have

Suppose the job is to make a release trustworthy. You know the outcome, but you do not yet know which workflow, script, test, or documentation page is authoritative. Start with the task as written:

codixing ask "release: verify assets and website" \
  --mode edit \
  --token-budget 4000

The colon is intentional. Task text is natural language, not a promise that the caller understands the underlying search grammar. v0.47.1 first tries the original query, then safely retries parser-significant punctuation as literal term boundaries if necessary. The pack keeps the original sentence as its task_summary.

The boundary matters: agents should describe intent in task language. Codixing is responsible for translating that intent into retrieval without making punctuation an accidental API.

Read the plan before opening files

ask returns a stable JSON context pack. The first useful distinction is between orientation and evidence: orientation explains where the task lives; evidence gives exact ranges that can be opened next.

{
  "task_summary": "release: verify assets and website",
  "mode": "edit",
  "repo_orientation": [
    {
      "path": "scripts/release_order.py",
      "why": "Contains evidence handle E1"
    }
  ],
  "must_read": [
    {
      "id": "E1",
      "path": "scripts/release_order.py",
      "range": "L46-L94"
    }
  ],
  "likely_tests": [
    { "path": "scripts/test_release_order.py" }
  ],
  "recommended_next_tools": [
    "read_file(E1)",
    "find_symbol(<symbol>)"
  ]
}

The response also carries related symbols, documentation, risks, changed-file anchors, and an estimated token total. The schema is deliberately predictable so an agent can route on fields instead of scraping an essay.

Follow evidence, then widen deliberately

The pack is not meant to dump the repository. It is a compact first pass. Follow the strongest evidence, jump to definitions when a symbol appears, and inspect blast radius before editing:

codixing context scripts/release_order.py --line 46 --token-budget 1000
codixing search IndexStore --strategy goto
codixing symbols IndexStore --defs-only
codixing impact .github/workflows/release.yml

Only widen when the compact answer proves insufficient. Both of the expensive views make that choice explicit:

codixing impact .github/workflows/release.yml --full
codixing graph . --map --token-budget 1500
Stage Command Decision it supports
orient ask Where does this task live, what evidence matters, and what should run next?
define search --strategy goto Which primary definition should the agent inspect before usages and tests?
trace impact What can break, and which tests are closest to the change?
widen impact --full Is the complete transitive blast radius necessary for this decision?
map graph --map How does the subsystem fit into the repository under a hard output budget?

Choose a mode that matches the work

For quick calls, ask can select a locate or understand shape automatically. An explicit mode makes the intent durable when the work is more specific:

  • locate for finding the implementation or owner of a behavior.
  • understand for assembling cross-file context before deciding what to change.
  • edit for implementation work with changed-file anchors and likely tests.
  • review for dependency-aware inspection of an existing change.
  • test for mapping behavior to the most relevant coverage.
  • migrate for changes that cross APIs, types, or file boundaries.
  • incident for fast operational orientation and risk discovery.

What the measured workflow saved

On this repository, the built-in bench-tokens harness measured a definition-and-blast-radius workflow at about 4,315 estimated tokens with Codixing versus 7,942 for its naive text-search-and-file-reading comparison. That is 3,627 fewer tokens, or 45.7%, for this specific workflow.

The important qualifier is “this workflow.” The harness is reproducible evidence, not a claim that every task saves the same percentage. Run it on the repository you care about:

codixing bench-tokens .

  /tmp/codixing-install.sh https://codixing.com/install.sh &&

Codixing v0.47.1 is available from GitHub Releases and as codixing-mcp@0.47.1 on npm. The full command reference is in the ask documentation.

The workflow is intentionally simple: state the task, inspect the plan, follow evidence, and widen only when the next decision requires it.
