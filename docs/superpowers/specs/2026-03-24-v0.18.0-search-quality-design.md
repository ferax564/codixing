## 1. Multi-Query Fusion for All Strategies

### Problem

Multi-query RRF fusion exists only in the `Deep` strategy (requires a reranker). The `Fast` and `Thorough` strategies — the most commonly used — run a single query and miss results that would surface under alternate phrasings.

### Design

#### 1.1 New public method: `Engine::search_multi`

```rust
/// Search with multiple query reformulations, fusing results via RRF.
///
/// Each query string is run through `search_first_pass()` independently.
/// Results are progressively fused using Reciprocal Rank Fusion.
pub fn search_multi(
    &self,
    queries: &[String],
    base_query: &SearchQuery,
) -> Result<Vec<SearchResult>>
```

Implementation:
1. For each query in `queries`, create a sub-query with the same `limit`, `file_filter`, and `token_budget` as `base_query`, with the reformulated query text.
2. Run `search_first_pass()` for each sub-query. Note: `search_first_pass` always runs hybrid retrieval (BM25+vector) with graph+definition+popularity boosts — it ignores the `strategy` field. This is the correct behavior: we want each reformulation to go through the same retrieval pipeline.
3. Progressive RRF fusion: fold result lists pairwise using `rrf_fuse()` with k=60 (same pattern as `search_deep` lines 640-643 — inline the fold, do not create a new utility function).
4. Truncate to `base_query.limit`.

#### 1.2 Auto-reformulation in Fast/Thorough strategies

In `Engine::search()`, for `Fast` and `Thorough` strategies, **only when the query is natural language (3+ words)**:
1. Call `generate_reformulations(&query.query)` (already exists, used by Deep).
2. If reformulations are non-empty, use `search_multi()` instead of the single-query path.
3. The original query is always included as the first query in the fusion.

**Gating:** Single-token identifier queries (e.g., `compute_pagerank`) and two-word queries (e.g., `IndexConfig new`) skip multi-query fusion entirely — the overhead of 2-4x retrieval passes is not justified when BM25 exact-match is already highly effective. This matches the existing `detect_strategy()` logic where 1-2 word queries use Instant/Fast without reformulation.

This means natural-language `Fast`/`Thorough` searches benefit from multi-query fusion, while identifier lookups remain fast.

#### 1.3 MCP tool: optional `queries` parameter

Add an optional `queries: Vec<String>` parameter to the `code_search` MCP tool. When provided:
- Skip auto-reformulation (the caller already provided explicit queries).
- Call `search_multi()` with the provided queries.
- This lets agents send their own multi-query reformulations.

### Files changed

- `crates/core/src/engine/search.rs` — new `search_multi()` method, modify `search()` for Fast/Thorough
- `crates/mcp/src/tools/search.rs` — add `queries` parameter to `code_search`

### Tests

- Unit test: `search_multi` with 3 queries returns fused results ranked higher than any single query
- Unit test: `search_multi` with overlapping results deduplicates correctly
- Integration test: `code_search` MCP tool with `queries` parameter

## 2. Query-Aware File Path Boosting

### Problem

`apply_path_match_boost()` handles dotted module paths (`django.db.models.lookups`) but misses explicit file path references (`path/to/file.py`) and backtick code references (`` `module/file.py` ``) commonly found in bug reports and agent queries.

### Design

Extend `apply_path_match_boost()` with two additional extraction patterns:

#### 2.1 Explicit file paths

Regex: `\b[a-zA-Z0-9_]+/[a-zA-Z0-9_/.-]*\.(py|rs|js|ts|go|java|rb|c|cpp|h|hpp|cs|swift|kt|scala|lua|m|sh|xml|yaml|yml|json|toml)\b`

The pattern requires at least one `/` to distinguish actual file paths from bare filenames like `error.py` that may appear in prose. The `\b` word boundaries prevent matching within larger tokens.

For each match, boost results whose `file_path` ends with the matched path segment. Boost factor: 2.5x.

#### 2.2 Backtick code references

Regex: `` `([^`]+)` `` — extract content between backticks, then:
- If it looks like a file path (contains `/` or `.ext`), apply file path boost (2.5x).
- If it looks like a dotted module path, apply the existing dotted-path logic.

#### 2.3 Integration

Both patterns are added to the existing `apply_path_match_boost()` function, which is already called by `PathMatchBoostStage` in the pipeline. No new pipeline stages needed.

### Files changed

- `crates/core/src/engine/search.rs` — extend `apply_path_match_boost()`

### Tests

- Unit test: file path `src/models/query.py` in query boosts matching result
- Unit test: backtick reference `` `django/db/models/lookups.py` `` boosts matching result
- Unit test: non-path backtick content (e.g., `` `QuerySet` ``) does not trigger path boost

## 3. Overlapping Chunks

### Problem

The cAST chunker produces non-overlapping chunks. When relevant context spans a chunk boundary (e.g., a type definition at the end of chunk A and a function using it at the start of chunk B), BM25 cannot match a query that requires both terms.

### Design

#### 3.1 Config

Add `overlap_ratio: f32` to `ChunkConfig`:

```rust
pub struct ChunkConfig {
    pub max_chars: usize,
    pub min_chars: usize,
    /// Fraction of overlap between adjacent chunks (0.0 = no overlap, 0.3 = 30%).
    /// When > 0, "bridge chunks" are generated at each chunk boundary.
    pub overlap_ratio: f32,
}
```

Default: `0.0` (backward compatible, no overlap).

#### 3.2 Bridge chunk generation

After the merge phase in `CastChunker::chunk()`, when `config.overlap_ratio > 0.0`:

1. Iterate over adjacent pairs of merged spans `(A, B)`.
2. Compute overlap size using float arithmetic: `overlap_bytes = ((A.size + B.size) as f32 / 2.0 * overlap_ratio) as usize`.
3. Create a bridge span: `byte_start = A.byte_end - overlap_bytes/2`, `byte_end = B.byte_start + overlap_bytes/2`.
4. Clamp to valid UTF-8 char boundaries using `str::is_char_boundary()`.
5. If the bridge span's non-ws size is >= `min_chars` and <= `max_chars`, emit it as an additional chunk.
6. Bridge chunks inherit the deeper scope chain from A or B.

Bridge chunks are interleaved with regular chunks in the output (sorted by `byte_start`). They receive distinct `chunk_id` values via the existing `chunk_id()` function (which hashes `file_path + byte_range`), so no special ID scheme is needed — the different byte ranges naturally produce unique IDs. Bridge chunks are indexed in Tantivy and the vector index alongside regular chunks.

#### 3.3 Deduplication impact

The existing `dedup_overlapping()` function removes results whose line ranges overlap with a higher-scored result from the same file. This naturally handles bridge chunks — if both a regular chunk and its bridge chunk match, only the higher-scored one survives.

### Files changed

- `crates/core/src/config.rs` — add `overlap_ratio` to `ChunkConfig`
- `crates/core/src/chunker/cast.rs` — bridge chunk generation after merge phase

### Tests

- Unit test: `overlap_ratio=0.0` produces same chunks as before (backward compat)
- Unit test: `overlap_ratio=0.3` produces bridge chunks at boundaries
- Unit test: bridge chunks have valid byte ranges (no mid-char splits)
- Unit test: bridge chunks are skipped when span is too small (< min_chars)

## 4. Git Recency Signal

### Problem

The search pipeline has no signal for file recency. Recently modified files are more likely to be relevant to active development concerns, but rank identically to files untouched for years.

### Design

#### 4.1 Recency map

Build a `HashMap<String, i64>` mapping relative file paths to their last commit timestamp. This is done via a single bounded `git log` invocation:

```bash
git log --format=%ct --name-only --diff-filter=ACMR --since="180 days ago" HEAD
```

The `--since="180 days ago"` bound matches the 180-day decay window in the boost formula, ensuring the git log is fast even on large repositories. Files not present in the output are treated as "older than 180 days" and receive no boost.

Parse the output: each commit timestamp line is followed by file paths. For each file, record the maximum (most recent) timestamp.

Store as `recency_map: OnceLock<HashMap<String, i64>>` on Engine (using `std::sync::OnceLock`). The `OnceLock` provides interior mutability for lazy initialization through shared `&self` references. The `search_context()` method calls `recency_map.get_or_init(|| build_recency_map(&self.store.root()))` to build the map on first search call, avoiding cold-start overhead for non-search operations.

#### 4.2 Pipeline stage: `RecencyBoostStage`

```rust
pub struct RecencyBoostStage;

impl SearchStage for RecencyBoostStage {
    fn apply(&self, results: &mut Vec<SearchResult>, ctx: &SearchContext<'_>) -> Result<()> {
        if let Some(recency_map) = ctx.recency_map {
            let now = SystemTime::now()
                .duration_since(UNIX_EPOCH)
                .unwrap()
                .as_secs() as i64;
            for r in results.iter_mut() {
                if let Some(&last_commit) = recency_map.get(&r.file_path) {
                    let days = (now - last_commit) / 86400;
                    let factor = (1.0 - days as f32 / 180.0).max(0.0);
                    r.score *= 1.0 + 0.1 * factor;
                }
            }
            sort_descending(results);
        }
        Ok(())
    }
}
```

#### 4.3 Pipeline integration

Add `RecencyBoostStage` to `fast_pipeline()` and `thorough_pipeline()`, after `PopularityBoostStage` and before `TestDemotionStage`.

#### 4.4 SearchContext extension

Add `recency_map: Option<&'a HashMap<String, i64>>` to `SearchContext`.

### Files changed

- `crates/core/src/engine/mod.rs` — add `recency_map` field, lazy initialization
- `crates/core/src/engine/pipeline.rs` — new `RecencyBoostStage`, add to pipelines
- `crates/core/src/engine/search.rs` — pass `recency_map` in `search_context()`

### Tests

- Unit test: `RecencyBoostStage` boosts recent files more than old files
- Unit test: files older than 180 days get no boost
- Unit test: missing files in recency map are unaffected
- Integration test: engine with git repo produces non-empty recency map

## 5. Tool Consolidation (Agent Experience)

### Problem

The MCP tool surface has discoverability issues:
- `stitch_context` has a non-obvious name that agents may never discover
- `search_usages` vs `symbol_callers` overlap is confusing — agents don't know which to use
- Note: `symbol_callers` and `symbol_callees` already use the `CodeGraph` (via `symbol_callers_precise`/`symbol_callees_precise` in `symbol_graph.rs`), falling back to BM25 only when the graph has no results. This is correct and does not need changes.

### Design

#### 5.1 Rename `stitch_context` → `assemble_context`

The MCP dispatch table is auto-generated from TOML files in `crates/mcp/tool_defs/`. To add the alias:
1. Add a new `[[tools]]` entry in `crates/mcp/tool_defs/search.toml` with `name = "assemble_context"` pointing to the same handler as `stitch_context`.
2. Keep the existing `stitch_context` entry for backward compatibility, but update its description to say: "Deprecated: use `assemble_context` instead."
3. Update the `assemble_context` description: "Assemble cross-file context by gathering related code from multiple files based on a query."

#### 5.2 Deprecation note on `search_usages`

Update the `search_usages` tool description in its TOML definition to add:
> "Tip: For graph-based caller discovery, use `symbol_callers` instead — it uses the dependency graph for more precise results."

### Files changed

- `crates/mcp/tool_defs/search.toml` — add `assemble_context` entry, update `stitch_context` description
- `crates/mcp/src/tools/search.rs` — add `call_assemble_context` handler (delegates to `call_stitch_context`)

### Tests

- Unit test: `assemble_context` dispatches to the same handler as `stitch_context`
- Unit test: `stitch_context` still works (backward compat)
- Verify tool descriptions in `search_tools` output
