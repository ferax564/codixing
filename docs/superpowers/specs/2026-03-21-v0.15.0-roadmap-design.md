# v0.15.0 Roadmap Design Spec

**Date:** 2026-03-21
**Version:** 0.14.0 → 0.15.0
**Status:** Approved

## Overview

9 work items across 5 branches in 3 dependency-aware waves. Focus areas: new languages, operational surface (HTTP server, federation management), distribution (VSIX), CI hardening, and release hygiene.

## Decisions Log

| Decision | Choice | Rationale |
|----------|--------|-----------|
| HTTP server scope | REST + SSE | SSE for long ops (reindex); no web UI yet |
| Federation management | CLI + MCP | Both humans and agents need access |
| Eval test `todo!()`s | Dropped | Test fixture data, not real gaps |
| Qdrant backend | Dropped | Speculative, no demand |
| VS Code extension | VSIX only | No marketplace overhead; attach to releases |
| Target version | 0.15.0 | Standard minor bump, save 1.0 for later |
| New languages | Matlab, Bash, Mermaid, XML/Draw.io | User-requested; covers code, scripting, diagrams, data |
| XML parsing | `quick-xml` crate, Config tier | No tree-sitter grammar; ConfigLanguageSupport trait |

## Wave Structure & Merge Order

```
Wave 1 — parallel (different sections of shared files):
  Branch A: feat/new-languages
  Branch B: chore/ci-improvements
  Branch C: chore/changelog-vsix

  Merge order: A → B → C
  (A first: touches Cargo.toml deps, resolve conflict surface earliest)

Wave 2 — after all Wave 1 merges:
  Branch D: feat/server-federation

  Rebase on main, then merge D.

Wave 3 — after Wave 2:
  Branch E: release/v0.15.0

  Final branch. Version bump + doc reconciliation.
```

Between each merge: pull main, verify CI passes (including GitHub Pages), then rebase next PR.

## Branch A: `feat/new-languages`

### Languages

#### Matlab (Tier 3 — full AST)
- **Crate:** `tree-sitter-matlab = "1.3"` (MIT, crates.io verified)
- **Extensions:** `.m` only (`.mlx` is a binary ZIP archive, not parseable as text)
- **Entities:** `function_definition`, `class_definition`, `properties`, `methods`, `enumeration`
- **Doc comments:** `%` line comments before definitions
- **Signatures:** `function [out1, out2] = name(in1, in2)`
- **Note:** `.m` extension conflicts with Objective-C. Add disambiguation: check for Objective-C markers (`#import`, `@interface`, `@implementation`) — if found, classify as Objective-C (not yet supported, skip file). Otherwise treat as Matlab.

#### Shell/Bash (Tier 3 — full AST)
- **Crate:** `tree-sitter-bash` (stable, widely used)
- **Extensions:** `.sh`, `.bash`, `.zsh`, `.bats`
- **Entities:** `function_definition`, exported `variable_assignment`, `alias`
- **Doc comments:** `#` comments before function definitions
- **Signatures:** `function_name()` or `function function_name`

#### Mermaid (Config tier — line-based parsing)
- **No tree-sitter crate available** (`tree-sitter-mermaid` is not published on crates.io; exists only as a GitHub repo with no stable releases)
- **Trait:** `ConfigLanguageSupport` (line-based parsing, same pattern as YAML/TOML)
- **Extensions:** `.mmd`, `.mermaid`
- **Symbols:** diagram type declarations (`flowchart`, `sequenceDiagram`, `classDiagram`, etc.), `subgraph` labels, node definitions with labels, `class` definitions
- **Doc comments:** `%%` comments
- **Parsing approach:** line-by-line regex extraction of diagram keywords, node definitions (`id[label]`, `id(label)`), and subgraph/class blocks. No dependency graph — diagrams are leaf documents.

#### XML/Draw.io (Config tier — custom parsing)
- **Crate:** `quick-xml` (workspace dep)
- **Extensions:** `.xml`, `.drawio`
- **Note:** `.svg` excluded — most SVGs are not diagrams and would produce noise. Draw.io SVGs can be renamed to `.drawio` if indexing is desired.
- **Trait:** `ConfigLanguageSupport` (same pattern as YAML, TOML, Dockerfile)
- **Symbols:** element names with `id`/`name`/`label` attributes; for `.drawio` files, extract `mxCell` `value` attributes as node labels and `mxGraphModel` diagram names
- **No tree-sitter dependency**

#### Route Implementations

All 13 routes are already wired in `routes/mod.rs`. Several handlers are implemented (search, graph); others need completion.

#### SSE Streaming for Reindex
- `/index/reindex` returns `Content-Type: text/event-stream`
- Events: `progress` (percentage + message), `result` (final stats), `error`
- Uses `axum::response::Sse` with `tokio_stream::wrappers::ReceiverStream`
- **New engine work required:** `Engine::sync()` currently returns `SyncStats` synchronously with no progress callback. Add a `sync_with_progress(callback)` method to `crates/core/src/engine/sync.rs` that accepts a channel sender, modeled after the MCP layer's `ProgressReporter` channel pattern (`crates/mcp/src/tools/common.rs:26`). The server's SSE handler will use a similar channel-based approach.

#### Request/Response Types
- Verify if existing route files already define inline types; add `crates/server/src/types.rs` only if shared types are needed
- `SearchRequest { query, limit, strategy, language_filter }`
- `SymbolRequest { name, kind, file_filter }`
- Reuse `codixing_core` types where possible

### Part 2: Federation CLI + MCP Tools

#### CLI Subcommands (`crates/cli/`)

```
codixing federation init                  # Create codixing-federation.json template
codixing federation add <name> <path>     # Add project to config
codixing federation remove <name>         # Remove project from config
codixing federation list                  # List projects + load status
codixing federation search <query>        # Federated search across all projects
```

#### MCP Tools (`crates/mcp/src/tools/federation.rs`)

| Tool | Description |
|------|-------------|
| `federation_init` | Create config template |
| `federation_add_project` | Add project by name + path |
| `federation_remove_project` | Remove project from config |
| `federation_list` | List projects, load status, file counts (subsumes existing `list_projects` tool) |
| `federation_search` | Federated search with RRF fusion |

**Note:** The existing `list_projects` tool (conditionally registered in `tool_definitions_with_federation()`) is subsumed by `federation_list`. Deprecate `list_projects` and redirect to `federation_list` for backward compatibility.

#### Core Additions (`crates/core/src/federation/`)
- `config_write()` — serialize config back to JSON
- `add_project()` / `remove_project()` — config mutation helpers (~30 lines each)
