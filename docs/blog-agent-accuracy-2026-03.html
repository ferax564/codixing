The benchmark nobody runs

Every code retrieval tool benchmarks search—can you find the right file? We already published those numbers. 100% top-1 accuracy, 99% token reduction, 24ms cold start.

But search accuracy isn't what makes agents fail. What makes agents fail is structural understanding—questions that require following dependencies, tracing call chains, and understanding what breaks when you change something.

These are the questions grep cannot answer:

  • "If I refactor this file, what breaks two hops away?"
  • "What is the cyclomatic complexity of each function in this module?"
  • "Which files are transitively affected through re-exports?"

We designed a benchmark that tests exactly these questions. Real agent sessions, real codebase, ground truth verified by hand.

How we tested

We used the Claude Agent SDK to run automated Claude Code sessions on OpenClaw—a 76-package pnpm monorepo with 9,387 indexed files.

Each task runs twice: once with vanilla Claude (grep, glob, read only) and once with Codixing tools available. Three runs per condition for statistical significance. Every task has a ground truth—a verified list of correct files or functions. We score recall: what fraction of the ground truth did the agent find?

Task 1: Transitive impact analysis

The question

"I want to refactor src/channels/plugins/target-parsing.ts. What files are affected transitively—not just direct importers, but files that import those files (2 hops deep)?"

Why grep fails

Grep can find import ... from "./target-parsing" in 4 files. But it can't trace the second hop—the files that import those 4 files. That requires a dependency graph. The full 2-hop blast radius is 12 files across src/ and extensions/.

Task 2: Cyclomatic complexity ranking

The question

"Rank these 3 files by cyclomatic complexity and name the most complex function in each: src/config/io.ts, src/gateway/server-methods/chat.ts, src/cron/service/jobs.ts."

Why grep fails

Grep cannot compute cyclomatic complexity. It can count keywords (if, for, while), but it can't scope those counts to individual functions without AST parsing. The vanilla agent tried reading the files and estimating, but consistently misidentified the most complex functions.

What this means

Search is solved. Structural understanding is not. AI agents with grep can find any symbol by name. That's table stakes. What they can't do is trace dependencies through re-exports, compute function complexity, or identify transitive blast radius. These are the tasks that determine whether a refactoring breaks production.

The gap is structural, not solvable by better prompting. No amount of prompt engineering will give grep a dependency graph. The vanilla agent used 95 tool calls on one transitive impact run, chaining grep after grep, and still missed 25% of affected files. Codixing used 6 calls and found 100%.

The efficiency gap compounds. 81% fewer tool calls means 81% fewer LLM round-trips. At $0.003/1K tokens for Sonnet, the token reduction saves roughly $0.015 per task. Over hundreds of agent sessions per day, that's real money—and real time.

Reproduce it

# Clone the benchmark
git clone https://github.com/ferax564/codixing
cd codixing

# Build
cargo build --release --workspace

# Run on OpenClaw (clones + indexes automatically)
python3 benchmarks/agent_benchmark.py \
  --repos openclaw --runs 3

The benchmark framework, task definitions, and ground truth are all in benchmarks/. The raw data from this post is in benchmarks/results/agent_benchmark.json.
