The problem with grep for AI agents

When Claude Code runs grep "Gateway" on OpenClaw—a 368K-line TypeScript monorepo—it gets 8,986 matches across 920 files (843 KB). The agent burns tokens parsing results it can't use, and still doesn't know which file defines the Gateway class.

It gets worse with conceptual queries. When an agent needs to find "how messages are routed between channels," grep returns 3,062 matches (325 KB) for "rout"—flooding context with imports, type references, and test mocks. We tested 8 domain-specific queries on OpenClaw:

How we got to 100%

On our Rust codebase (138 files, a fair test since we control the code), BM25-only search started at 70% top-1 accuracy. Three misses exposed fundamental retrieval gaps:

Miss 1: "dead code detection" → expected orphans.rs

BM25 couldn't bridge the vocabulary gap. The word "dead" never appears in orphans.rs. We added concept-to-path boosting—a mapping from concept terms to file path patterns. "dead code" → orphan/, "reciprocal rank fusion" → hybrid/, "tree-sitter" → parser/. Six mappings, plus a stem-based fallback that matches "parsing" to parser/ by comparing 5-character prefixes.

Miss 2: "asymmetric RRF" → expected hybrid.rs

hybrid.rs was being demoted by the search infrastructure filter. We had a blanket 0.5× penalty on engine/search.rs and retriever/hybrid.rs because they're self-referential—their synonym maps mention every search concept. But when the query IS about search concepts ("rrf", "fusion", "hybrid"), the demotion is wrong. Fix: query-aware demotion bypass.

Miss 3: "tree-sitter AST" → expected parser files

BM25 found files that use tree-sitter (like sync.rs) rather than files that are the parser. The stem-based path matching ("tree" → tree_cache/) solved this alongside concept-to-path boost.

Session boost: from additive to multiplicative

During multi-query sessions, accumulated session boosts caused files touched by earlier queries to override later results. We switched from additive (score += boost) to multiplicative capped at 1.3× (score *= 1 + min(boost, 0.3)), preserving relative ranking from the retrieval pipeline.

What matters for AI agents

These benchmarks highlight three things that matter in practice:

1. Token budget is the real constraint. An agent with a 200K context window can afford 6 KB of search results, not 843 KB. On OpenClaw, grep returns an average of 334 KB per query—Codixing returns 6 KB. The 99% token reduction directly translates to more productive agent turns.

2. Grep drowns agents in noise at scale. On a 7,607-file monorepo, even a simple query like "Gateway" produces 8,986 matches across 920 files. The agent can't parse 843 KB of results—it needs 10 ranked results that point to the right files. That's what Codixing delivers.

3. Cold start determines responsiveness. At 50ms for a 7,607-file codebase, Codixing adds negligible latency to agent tool calls. Even on the Linux kernel (73K files, 30M lines), cold-start search completes in 1.57 seconds—fast enough for interactive agent use. A 500ms+ startup from a Node.js-based MCP tool creates a perceptible delay on every new session.
