v0.43.0: safer MCP agents, profile-gated tools, and doctor diagnostics

Codixing v0.43.0 is a hardening release for AI agents working inside large repositories. The MCP server now reports its real package version, defaults toward read-only workflows, gates write tools behind explicit profiles, isolates daemon sockets by profile, and ships a new codixing doctor command for debugging local setup.

Why this release matters

Codixing already had a strong retrieval core: Tree-sitter parsing, BM25, vectors, symbol indexes, dependency graphs, MCP, CLI, LSP, HTTP, and daemon mode. The weak point was not search quality. It was operational discipline around agents.

When an editor connects an AI agent to a repository, the safest default should be simple: read first, write only when the user explicitly opts in. v0.43.0 moves Codixing in that direction.

The design goal: make Codixing feel less like a pile of raw tools and more like a repository control plane that can keep agents focused, observable, and safer by default.

MCP profiles

The MCP server now supports profile-scoped tool exposure. Clients can request a narrower surface instead of receiving every available tool in every context.

Profile Purpose
minimal Search, symbols, repo map, schema discovery, and index status.
reviewer Read-only default for review and analysis workflows.
editor Read tools plus non-destructive write helpers, enabled explicitly.
dangerous The full tool surface, including destructive operations, for deliberate local use. 
codixing-mcp --root . --profile minimal
codixing-mcp --root . --profile reviewer
codixing-mcp --root . --profile editor --allow-write-tools

tools/list now reflects the selected profile, and tools/call blocks known tools that the active profile does not allow. The daemon endpoint is profile-scoped too, so a narrower client does not accidentally proxy through a broader daemon socket.

Accurate MCP server identity

The MCP initialize response no longer carries a stale hardcoded version. It now reports the package version from Cargo metadata, so clients and support traces see the same version that was built and released.

serverInfo.version = 0.43.0

This is small but important. Agent tooling is easier to debug when the server tells the truth about itself.

codixing doctor

v0.43.0 adds codixing doctor, a local diagnostic command for setup and support. It reports the binary version, index health, stale index status, model/runtime availability, disk usage, locks, daemon status, ignored-file summary, embedding progress, and benchmark reproducibility details.

codixing doctor

The point is to make "why is my agent slow, stale, or disconnected?" answerable without spelunking through index directories and daemon logs.

Release validation

The release went through the full local and GitHub path:

  • cargo test --workspace passed locally.
  • cargo clippy --workspace -- -D warnings passed locally.
  • cargo fmt --check passed locally.
  • PR CI passed on Ubuntu, macOS, and Windows.
  • Main CI produced Linux x86_64, macOS aarch64, and Windows x86_64 release binaries.
  • The GitHub release and npm publish workflows completed successfully.

The release is available on GitHub Releases, and the npm package is published as codixing-mcp@0.43.0.

What comes next

The next focused feature is agent_context_pack: a stable context compiler that can return ranked repo orientation, must-read file ranges, related symbols, likely tests, docs, risks, and recommended next tools under a token budget.

That is the next step toward the larger product direction: Codixing as the repository control plane for AI software agents.
