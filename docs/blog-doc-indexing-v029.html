Your docs are invisible to code search. We fixed that.

Codixing now indexes markdown and HTML documentation alongside source code. One search box, unified results, doc-to-code linking — all running locally in Rust.

The problem

You search for "rate limiting" in your codebase. You get the RateLimiter struct, the middleware implementation, maybe a test. But that architecture doc your teammate wrote explaining why the algorithm was chosen? Invisible. The README section explaining how to configure rate limits? Gone.

Every code search tool I've used treats documentation as a second-class citizen. Source code lives in the index. Docs live in... hope you remember which file it was in.

Starting with v0.29.0, Codixing treats .md and .html files as first-class searchable content. No separate command, no separate index, no configuration. Just codixing init and your docs are in.

How it works: a walkthrough

Let's say your repo has this README:

# My Project

## Getting Started

Call `Engine::init()` to initialize the search engine.
Configure via `IndexConfig`.

## Installation

### Linux

Run the install script:
```bash
curl -fsSL https://codixing.dev/install.sh | bash
```

After codixing init, search "how to install":

$ codixing search "how to install"

1. README.md § My Project > Installation > Linux [doc] score=4.231
   | Run the install script:
   | ```bash
   | curl -fsSL https://codixing.dev/install.sh | bash

2. docs/install.sh [L1-L45] (Bash) score=2.107
   | #!/bin/bash
   | set -euo pipefail

Notice two things:

  • The breadcrumb — § My Project > Installation > Linux tells you exactly where in the doc this result lives, without line numbers.
  • Both docs and code appear together — the README section ranks first (it's the most relevant), the install script ranks second. One search, both worlds.

Section-aware chunking

Codixing doesn't dump your entire README into one blob. It parses the heading hierarchy — #, ##, ### — and creates chunks at section boundaries, each carrying a section breadcrumb path.

The algorithm mirrors our code chunker (cAST) with three phases:

  1. Section split — each heading starts a new chunk. If a section is too large, it splits at paragraph boundaries (\n\n).
  2. Atomic preservation — code blocks and tables are never split mid-element.
  3. Merge pass — adjacent tiny sections are merged to avoid index bloat.

The breadcrumb path (e.g., ["Getting Started", "Installation", "Linux"]) is stored in the same scope_chain field that code uses for AST scope paths like ["my_module", "MyClass", "my_method"]. No schema change needed — the semantics are identical: "where am I in the hierarchy."

Doc-to-code linking

When your docs mention code symbols in backticks, Codixing notices. If your README says `Engine::init()` and that function exists in your codebase, a DocumentedBy graph edge is created.

This means codixing callers src/engine.rs now shows both:

  • Code files that import/call functions in engine.rs
  • Doc files that reference symbols defined in engine.rs

The linking is conservative — only unambiguous symbol matches (exactly one defining file) create edges. Common names like new or parse are skipped.

Query routing

When you search for ChunkConfig (CamelCase, clearly a symbol), you probably want code results. When you search for "how does rate limiting work" (natural language), you probably want docs. Codixing detects this automatically.

A lightweight heuristic classifier runs on every query:

  • Code signals: CamelCase, snake_case, ::, (), fn, def
  • Doc signals: "how", "what", "why", "guide", "tutorial"
  • Mixed or ambiguous: equal weights (the default)

When the intent is detected, off-target results get a 0.3x score dampening. Docs still appear for code queries if they're highly relevant — it's a soft boost, not a hard gate.

For explicit control, use the new flags:

$ codixing search "rate limiting" --docs-only
$ codixing search "RateLimiter" --code-only

What we didn't build (and why)

The RAG research for this feature surfaced a lot of techniques. Here's what we deliberately left out:

  • Contextual retrieval (LLM at index time) — prepending LLM-generated context to chunks improves retrieval by ~35% per Anthropic's research, but requires an LLM call per chunk at indexing time. Codixing is local-first, no API keys.
  • HyDE (hypothetical document embeddings) — generates a synthetic "ideal answer" per query. Adds seconds of LLM latency to every search. Not compatible with sub-millisecond BM25.
  • Semantic chunking — splits at embedding similarity boundaries. Benchmarks show it underperforms recursive splitting (54% vs 69% end-to-end accuracy). The "better retrieval recall" doesn't translate to "better answers."
  • Parent-child dual storage — stores both parent sections and child chunks. Doubles index size for a quality gain we can add later. The section breadcrumb captures 80% of the benefit.

Under the hood

Two new Rust crates do the heavy lifting:

  • comrak (Markdown) — the same parser that powers crates.io and docs.rs. Produces an Arena-allocated AST we walk for heading hierarchy, code blocks, and element detection.
  • scraper (HTML) — built on html5ever, handles h1-h6 sections with CSS selectors. Extracts <code> tag contents for symbol linking.

The architecture is intentionally simple: doc files go through the same indexing pipeline as code. They end up in the same Tantivy index, the same vector index, the same graph. One new field (doc_type) distinguishes them. One new trait (DocLanguageSupport) handles parsing. Everything else is reused.

Backward compatible: If you upgrade Codixing and open an existing index, it works fine. The doc_type field is Option<Field> — old indexes just don't have it. Run codixing init when you're ready to index docs.

What's next

External doc crawling is the obvious Phase 2. Imagine codixing add-docs https://docs.rs/tantivy indexing your dependency's documentation alongside your code. The architecture already supports it — the HTML parser and doc_type field are ready. We just need a crawl/fetch layer.

For now, in-repo docs cover the most impactful case: the READMEs, architecture docs, and API guides that developers actually write and forget to search.
