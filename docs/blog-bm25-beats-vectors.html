How we got BM25 to outperform vector search for code retrieval

We spent a week testing 6 embedding approaches for our code search engine. Static lookups, int8 transformers, code-specific distillation. None of them beat our BM25 pipeline. Here's the data and what we learned.

The setup

Codixing is a code retrieval engine for AI agents. It indexes a codebase and answers queries like "rate limiting implementation" or "cron scheduling" by returning the most relevant files.

Our benchmark uses OpenClaw — a real 9,400-file TypeScript codebase — with 5 concept queries, each with manually curated ground-truth files. We measure R@10 (what fraction of relevant files appear in the top 10 results) and MRR (how high the first relevant result ranks).

BM25 wins on every metric. The best embedding model (BGE-Small-Q) gets close but takes 175x longer to index.

Why embeddings didn't help

1. BM25 was already doing the semantic work

Our BM25 pipeline isn't vanilla BM25. It has:

  • BM25F weighted fields — doc comments get 2x boost, identifier words get 2x, path segments get 1.5x
  • Code-stemmed tokenizer — English stemming applied to concept fields so "scheduling" matches "schedule"
  • 80 synonym groups — "rate limiting" expands to include "throttle", "quota", etc.
  • Multi-query RRF — queries with 3+ words generate synonym expansions and code-HyDE reformulations, each searched independently, then fused with Reciprocal Rank Fusion

This combination already bridges the vocabulary gap that embeddings are supposed to solve. When someone searches "cron scheduling", synonym expansion adds "schedule" and "periodic", and BM25F fields match against identifier words. There's nothing left for the vector signal to contribute.

2. Vector results dilute good BM25 rankings

The hybrid search uses asymmetric RRF — for natural language queries, vector results get higher weight. But when BM25 already found the right files and the vector index returns noisy near-neighbors from 47K chunks, the fusion pulls irrelevant files into the top 10.

We saw this directly: for short queries like "rate limiting", BM25 alone puts auth-rate-limit.ts at #1 (MRR=1.0). With vectors, it drops to #2 because a semantically similar but irrelevant file gets promoted.

3. Static embeddings can't handle code

Model2Vec uses a BERT WordPiece tokenizer trained on English prose. Code identifiers get shredded:

composeSystemPrompt → compose##sy##ste##mp##rom##pt  (5 of 6 tokens meaningless)
parseCronSchedule   → par##se##cr##ons##ched##ule  (all fragments)

We built a preprocessing step that splits CamelCase/snake_case before tokenization, reducing subword fragments by 50-70%. We even distilled a code-specific Model2Vec from Jina Code — the BPE tokenizer handles CamelCase perfectly (createRateLimiter → create, Rate, Limiter). But both performed the same or worse in the actual pipeline because BM25 was already there.

4. Transformers help quality but kill indexing speed

BGE-Small-Q (the best embedding model) achieves R@10=0.447 — close to BM25's 0.513 but not better. And it takes 70 minutes to index what BM25 does in 24 seconds. Codixing is designed for instant search after codixing init. A 70-minute embedding step would destroy the experience.

What actually worked

Instead of embeddings, the features that moved the needle were all BM25-side improvements:

  1. BM25F concept fields with doc_comment and identifier_words boosting (+11.8% concept R@10)
  2. Synonym expansion in Fast/Thorough strategies — moved from Deep-only
  3. Code-stemmed tokenizer — English stemming on concept-oriented fields
  4. Graph semantic propagation — inject callers/callees of top results post-retrieval
  5. File-level dedup with allowance — prevent same-file chunk flooding while preserving struct+impl pairs

These are all deterministic, zero-latency improvements. No model to download, no embedding to compute, no ONNX runtime to configure.

When embeddings would help

Our BM25 pipeline works because it's been heavily tuned for code. Embeddings would likely win in:

  • Multilingual code — BM25 synonyms are English-only; embeddings work cross-language
  • No synonym expansion — vanilla BM25 without our 80 synonym groups would benefit from semantic vectors
  • Query-document language gap — "find the authentication middleware" searching code that only uses "auth" in identifiers

The takeaway

BM25 is underrated. If you're building a code search system and reaching for embeddings, consider investing that effort in BM25 tuning instead: code-aware tokenization, synonym tables, field boosting, multi-query RRF. You might find it already does what you thought you needed vectors for. 175x faster.
