Installation

Codixing ships as pre-built binaries for Apple Silicon macOS, Linux x86_64, and Windows x86_64. No Rust toolchain or compilation is required.

One-line install (recommended)

Downloads the checksum-verified four-binary suite. It uses /usr/local/bin when writable and otherwise $HOME/.local/bin; override with CODIXING_INSTALL_DIR and pin with CODIXING_VERSION=X.Y.Z.

shell
curl --proto '=https' --proto-redir '=https' -fsSLo /tmp/codixing-install.sh https://codixing.com/install.sh
sh /tmp/codixing-install.sh

Installs four binaries: codixing (CLI), codixing-mcp (MCP server), codixing-lsp (LSP server), and codixing-server (HTTP/SSE server).

ℹ The shell installer supports Apple Silicon macOS and Linux x86_64. Windows x86_64 binaries are available on the releases page; npx -y codixing-mcp is the portable MCP-only option.

Build from source

If you prefer to compile yourself, install Rust 1.88 or newer via rustup.

shell
# Clone the repository
git clone https://github.com/ferax564/codixing
cd codixing

# Build all binaries with release optimisations
cargo build --release --workspace

# Binaries are now at:
./target/release/codixing       # CLI
./target/release/codixing-mcp   # MCP server
./target/release/codixing-lsp   # LSP server
./target/release/codixing-server # HTTP/SSE server

Optional features

Feature flagWhat it enables
--features qdrantExperimental Qdrant client APIs; the CLI does not yet replace its built-in vector backend with Qdrant
--features qwen3Qwen3-Embedding-0.6B via ONNX Runtime (1024 dimensions)
ℹ The default build can use BGE embeddings through fastembed, but ordinary codixing init . remains BM25-only. BGE requires --embed plus ONNX Runtime; static model2vec embeddings do not require ONNX or an external model server.

Quick Start

Index a repository, search it, and wire it up to Claude Code in under a minute.

shell
# 1. Index your project (BM25 only — instant, no model download)
codixing init .
✓ Indexed 138 files, 1113 chunks, 2323 symbols in 0.21s

# 2. Search with natural language
codixing search "authentication handler"
► src/auth/handler.rs:42  [score: 0.94]
pub fn handle_auth_request(req: Request) -> Result<Token>

# 3. Or with full hybrid (downloads BGE model on first run)
codixing init . --embed --model bge-small-en
codixing search "authentication handler" --strategy fast

CLI — init

Builds a Codixing index inside the .codixing/ directory at the given path. Use sync or git-sync for routine incremental updates; use init when you intentionally want to reconstruct the complete index.

usage
codixing init [OPTIONS] [PATH]
FlagDefaultDescription
--embedfalseEnable vector embeddings. BM25-only is the fast default.
--model <name>bge-base-enSelect the model used with --embed. Supported families include BGE, Jina Code, Nomic, Snowflake Arctic, Qwen3 (feature-gated), and the ONNX-free model2vec variants; run codixing init --help for the accepted names.
--threads <N>min(CPUs, 8); min(CPUs, 4) on WindowsOverride the bounded indexing worker count. Start with the default and change it only after measuring on your repository and storage.
--max-file-bytes <BYTES>2 MiBSkip larger files during indexing to bound work and memory. Use 0 only when an unlimited file size is intentional.
--also <DIR>—Add an extra root directory to the same index (multi-repo mode).
examples
# Fast BM25-only index of current directory
codixing init .

# Full hybrid index (waits for a durable vector checkpoint)
codixing init /path/to/project --embed --model bge-small-en

# Multi-repo: index two roots together
codixing init . --also ../shared-lib

CLI — search

Query the index with a natural language phrase or symbol name. Depending on the selected strategy, retrieval uses lexical, vector, graph, reranker, exact, or behavioral-signature stages; MMR deduplication is applied only by strategies that request it.

usage
codixing search <QUERY> [OPTIONS]
FlagDefaultDescription
--strategy <S>autoOne of: auto, instant, fast, thorough, explore, deep, exact, semantic
--limit <N>10Maximum number of results to return
--token-budget <N>—Token budget for formatted output; implies --format
--file <PATH>—Restrict results to matching file paths
--code-only / --docs-onlyfalseRestrict results by source type
examples
# Natural language query, default strategy
codixing search "parse JWT token"

# Identifier query — auto-routes to BM25-dominant
codixing search "TokenVerifier"

# Thorough hybrid with custom token budget
codixing search "database connection pool" --strategy thorough --token-budget 4000

# Cross-encoder reranking (configure it while indexing)
codixing init . --embed --reranker
codixing search "error handling middleware" --strategy deep

CLI — symbols

Look up definitions (functions, structs, classes, methods) by name. Case-insensitive substring match across all indexed files.

usage

CLI — callers / callees

Traverse the import and call graph to find who depends on a file, or what a file depends on.

FlagDefaultDescription

CLI — agent-context-pack

Compile a stable JSON context pack for an AI agent task. The pack includes repo orientation, must-read evidence handles, related symbols, likely tests, docs, risks, and recommended next tools.

usage
codixing agent-context-pack <TASK> [--mode edit] [--changed-file <PATH>]
FlagDefaultDescription
--mode <M>understandOne of: locate, understand, edit, review, test, migrate, incident
--token-budget <N>6000Approximate maximum tokens for the JSON payload
--changed-file <PATH>—Repeatable task-local file anchor pinned into the pack
--branch <NAME>—Optional branch/worktree label copied into the schema
--risk-level <LEVEL>—Optional caller risk label such as low, medium, or high
example
codixing agent-context-pack "add branch-aware artifact cache" \
  --mode edit \
  --changed-file crates/core/src/catalog.rs \
  --token-budget 6000

CLI — sync & git-sync

sync

Hash-based incremental sync. Walks all files, hashes content (xxh3), and only reindexes files that have changed or been deleted since the last run.

usage
codixing sync [PATH]

git-sync

Git-aware incremental sync. Uses git diff --name-status between the last indexed commit and HEAD. Faster than sync on large repos since it only processes files reported by git.

usage
codixing git-sync [PATH]

GitSyncStats { changed: 3, deleted: 1, unchanged: 766 }
Sync complete in 0.12s
ℹ git-sync returns a no-op when Git metadata or the required diff is unavailable. Use sync for non-Git projects and whenever uncommitted/untracked filesystem drift must be authoritative.

CLI — repository analysis workflow

Compose the supported commands below to index a repository, inspect its architecture, generate a self-contained graph dashboard, and check the blast radius of a file.

usage
git clone --depth=1 https://github.com/owner/repo.git
cd repo
codixing init .
codixing graph . --map --token-budget 8000
codixing graph . --html graph.html
codixing impact --json src/path/to/file.rs

# Optional local graph/API server
codixing serve . --port 8080
ℹ init rebuilds the complete index. For routine updates after the first index, use sync or git-sync.

Search Strategies

codixing search and the MCP code_search tool accept a strategy parameter. The default auto route selects an appropriate strategy from the query and available index capabilities.

For a one-shot CLI query on a very large repository, explicitly selecting instant or exact opens a lean lexical read profile that skips graph, vector, and reranker loading. The auto route and every other strategy open the full index so their ranking features remain available.

instant
BM25 only. No vector scoring, no graph boost.
fastest · zero model required
fast
BM25 + vector RRF fusion + graph PageRank boost.
requires vectors for hybrid scoring
thorough
Same as fast but with MMR deduplication applied.
higher diversity
explore
BM25 first-pass then graph neighbour expansion (RepoHyper-style).
best for architecture queries
deep
thorough + BGE-Reranker-Base cross-encoder second pass.
requires the reranker configured at init
exact
Trigram substring lookup with deterministic bounded ranking.
best for exact identifiers and literals
semantic
Embedding-free matching over behavioral signatures.
concept matching without ONNX

Asymmetric routing

Codixing automatically detects whether a query looks like an identifier (e.g. TokenVerifier, b2Vec2) and adjusts the RRF weights accordingly:

  • Identifier queries → BM25-dominant (k=20/90) — exact token match matters more
  • Natural language queries → vector-dominant (k=90/20) — semantic similarity matters more

Hybrid Retrieval

BM25

Tantivy-powered full-text index with a custom CodeTokenizer that splits on camelCase, snake_case, and punctuation. Field boosts ensure definitions outrank mentions:

  • signature field: ×3.0 boost
  • entity_names field: ×2.0 boost
  • content field: ×1.0

Vector

BGE-Base-EN-v1.5 (768-dim) embeddings via fastembed ONNX, stored in a usearch HNSW index with int8 quantisation (8× memory reduction). Chunks are embedded with contextual prefixes for better recall:

contextual prefix format
"{file_path} | {scope_chain} | {signature}\n{content}"

Reciprocal Rank Fusion (RRF)

BM25 and vector results are merged with RRF. The formula assigns a score to each candidate based on its rank in each list:

score(d) = Σ  1 / (k + rank_i(d))

Band merging (LDAR)

Adjacent chunks from the same file within 3 lines of each other are merged before output. This reduces token count by 25–91% on high-hit queries without losing context.

MCP — Setup

The codixing-mcp binary implements JSON-RPC 2.0 over stdin/stdout and is compatible with Claude Code and any MCP-capable client.

Default mode

The first invocation normally auto-starts a background daemon and proxies the MCP connection to it. Pass --no-daemon-fork when a client should own a direct stdio process instead.

shell
codixing-mcp --root /path/to/project

Daemon mode

The daemon holds the engine in memory and serves subsequent calls over a profile-specific Unix socket (.codixing/daemon-<profile>.sock; escalation-enabled policies use a separate suffix). This avoids reloading a large index for each connection.

shell
codixing-mcp --root /path/to/project --daemon

Auto-init

If no index exists at --root, the MCP server automatically builds a BM25-only index before serving the first request. For large repositories, run codixing init . ahead of time so client startup timeouts do not interrupt indexing.

Runtime tool profiles

minimal is the default and exposes the smallest read-only discovery set. Agents can call get_mcp_profile and set_mcp_profile to switch within the process's startup safety ceiling. Minimal/reviewer startup cannot promote itself into a write-capable profile unless --allow-profile-escalation was supplied explicitly; dangerous also requires per-call confirmation. Successful switches emit notifications/tools/list_changed so clients can refresh tools/list.

Every tool result passes through a 4,000-token default response envelope with a 12,000-token hard maximum. Tool-specific limits may be lower, and large result sets should be paged rather than requesting one unbounded response. Individual JSON-RPC documents are capped at 8 MiB.

VS Code / Cursor extension

Install the extension from editors/vscode/. Its command-palette actions can index, sync, search, inspect the repository, start the daemon, and write the MCP server entry to both ~/.claude.json and ~/.cursor/mcp.json.

MCP — Selected tools reference

All tools are called via JSON-RPC 2.0 tools/call with a name and arguments object. tools/list returns only the active profile's subset; use search_tools and get_tool_schema for compact discovery. Write tools automatically reindex after an operation.

code_search Strategy-routed ranked search

The primary retrieval tool. Routes across BM25, optional vectors, graph expansion, reranking, exact trigram lookup, and behavioral signatures according to strategy; stages that are unavailable or unnecessary are skipped.

parameters
{
  "query":    "authentication middleware",  // required
  "strategy": "fast",                     // instant|fast|thorough|explore|deep|exact|semantic
  "limit":    20,
  "token_budget": 4000
}
find_symbol Case-insensitive definition lookup

Searches the symbol table for definitions matching the given name. Returns file path, line number, kind, and signature.

parameters
{
  "name": "HybridRetriever",
  "file": "retriever"  // optional path substring
}
read_symbol Source block for a symbol by name

Returns the resolved source block for a symbol (function body, struct definition, class, etc.), subject to the response token budget.

parameters
{
  "name": "handle_auth_request",
  "file": "src/auth"  // optional disambiguation
}
get_repo_map Token-budgeted PageRank architecture map

Returns an Aider-style repository map: files sorted by PageRank with their top-level symbol signatures. Excellent for giving an agent a high-level orientation of a codebase.

parameters
{
  "token_budget": 4000
}
get_references Callers + callees for any file

Returns the direct import/call graph neighbours of a file — who imports it, and what it imports.

parameters
{
  "file": "src/auth/handler.rs"
}
get_transitive_deps Multi-hop dependency traversal

Traverses the files a target depends on up to the specified depth. Use predict_impact for the dependent/blast-radius direction.

parameters
{
  "file":  "src/index/bm25.rs",
  "depth": 3
}
search_usages Ranked or deterministic paged usages

Returns a ranked view by default. Set complete=true for a deterministic, pageable call-site/import scan and follow next_offset until has_more=false. Complete mode scans at most 100,000 candidate chunks, so completion is relative to that ceiling.

parameters
{
  "symbol":   "PageRank",
  "complete": true,
  "limit":    20,
  "offset":   0
}
grep_code Regex/literal search with context lines

Token-budgeted grep with trigram pre-filtering, match highlighting, context lines, and glob filtering. A file-level trigram index skips non-candidate files before any disk I/O (Russ Cox/trigrep technique). Supports full regex patterns via HIR-walking query plans with OR-branch union. Designed for LLM consumption — far more efficient than raw rg output.

parameters
{
  "pattern":  "fn handle_\\w+",
  "file_glob": "**/*.rs",   // optional file filter
  "context_lines": 2,        // hard max: 5
  "literal": false
}
read_file Raw source with optional line range

Returns the raw source of a file, optionally sliced to a line range. Respects the token budget.

parameters
{
  "file":         "src/engine.rs",
  "line_start":   80,   // 0-indexed, optional
  "line_end":     140,  // 0-indexed, optional
  "token_budget": 4000
}
index_status Diagnostic report with full index stats

Returns a snapshot of the current index state: file count, symbol count, vector count, graph node/edge counts, last sync commit, and strategy availability.

parameters
{}  // no parameters required
get_mcp_profile Inspect the active MCP tool profile

Returns versioned JSON with the active profile for the current connection, the available profiles, and their safety levels. Use this before requesting broader tool access.

parameters
{}  // no parameters required
set_mcp_profile Switch tools without restarting MCP

Changes the active profile within the process's startup ceiling and emits notifications/tools/list_changed. A minimal/reviewer process needs --allow-profile-escalation before it can enter a write profile; dangerous also requires explicit confirmation.

parameters
{
  "profile": "editor",           // minimal|reviewer|editor|dangerous
  "confirm_dangerous": false   // true required for dangerous
}
write_file Write a file and reindex it immediately

Creates or overwrites a file at the given path (relative to project root) with the provided content, then automatically reindexes it so the new symbols and content are immediately searchable. Path traversal outside the project root is blocked.

parameters
{
  "file":    "src/auth/middleware.rs",  // relative path
  "content": "pub fn verify_token(...) { ... }"
}
edit_file Replace a unique string in a file and reindex

Performs an exact string replacement in a file: finds old_string, replaces it with new_string, writes the result, and reindexes. Returns an error if old_string appears 0 times (not found) or more than once (ambiguous) — this guarantees surgical edits with no unintended side effects.

parameters
{
  "file":       "src/auth/middleware.rs",
  "old_string": "pub fn verify_token(token: &str)",
  "new_string": "pub fn verify_token(token: &str, strict: bool)"
}
delete_file Delete a file and remove it from the index

Deletes a file from disk and removes all associated chunks, symbols, vectors, and graph edges from the live index. The change is persisted immediately so subsequent searches will not return stale results from the deleted file.

parameters
{
  "file": "src/auth/old_handler.rs"  // relative path
}
list_files Page indexed files with optional glob filtering

Returns a bounded, lexicographically sorted page of files tracked in the Codixing index, with chunk counts per file (a proxy for size). Use before bulk edits or to explore project structure without consuming tokens on a full repo map.

parameters
{
  "pattern": "src/**/*.rs",  // optional glob
  "limit":   100              // default and hard maximum
}
rename_symbol Plain substring replacement across indexed files

This is not a semantic rename: it replaces the old_name substring even inside larger identifiers, comments, and strings, then re-indexes modified files. Optional file_filter restricts matching paths. Use dry_run=true first, and prefer an LSP-backed rename for ambiguous identifiers.

parameters
{
  "old_name":    "compute_pagerank",
  "new_name":    "rank_nodes",
  "file_filter": "crates/core"  // optional path substring filter
}
explain Assembled understanding package for a symbol

Produces a single LLM-ready context block combining: the symbol's full source definition, the dependency graph for its containing file (what imports it, what it imports), and the top call sites from the index. Ideal as a first step before modifying any significant function or class.

parameters
{
  "symbol": "compute_pagerank",
  "file":   "graph/mod.rs"  // optional disambiguator
}
agent_context_pack Stable JSON context pack for agent workflows

Compiles ranked repo orientation, must-read evidence handles, related symbols, likely tests, docs and conventions, risks, and recommended next tools under a token budget. Use as the first call for locate, understand, edit, review, test, migrate, and incident workflows.

parameters
{
  "task":          "add branch-aware artifact cache",
  "mode":          "edit",
  "token_budget":  6000,
  "changed_files": ["crates/core/src/catalog.rs"],
  "risk_level":    "high"
}
outline_file Per-file symbol tree sorted by line number

Returns a line-sorted symbol outline for one file (functions, classes, structs, methods, etc.) within the response envelope. More token-efficient than read_file when you need to understand a file's structure before diving into a specific function. Includes line ranges, kinds, and signatures.

parameters
{
  "file": "src/engine.rs"  // relative path
}
apply_patch Apply a unified git diff with auto-reindex

Accepts a unified diff in standard git format (--- a/file / +++ b/file / @@ hunks @@), applies its file hunks, and re-indexes modified files. A multi-file patch can report partial errors, so inspect the result before treating the operation as complete.

parameters
{
  "patch": "--- a/src/foo.rs\n+++ b/src/foo.rs\n@@ -1,3 +1,4 @@\n context\n+new line\n more"
}
run_tests Execute a test command and return the output

Runs a shell command string via sh -c (or cmd /C on Windows) inside the project root and returns bounded output with the exit code. Each stream has a capture ceiling; the combined response keeps at most the most recent 8,000 characters before the final MCP response envelope. Default timeout: 120s.

parameters
{
  "command":      "cargo test my_test",
  "timeout_secs": 120  // default
}
symbol_callers Bounded direct callers for a symbol

Returns up to limit direct callers from the symbol-level call graph, with a search fallback when the graph has no result. Use paged search_usages complete=true for a broader deterministic scan.

parameters
{
  "symbol": "process_file"
}
symbol_callees Bounded direct callees for a symbol

Returns up to limit direct callees for the named symbol. Complements symbol_callers for bidirectional call-graph inspection.

parameters
{
  "symbol": "process_file"
}
predict_impact Rank files by blast-radius impact from a diff

Parses a unified diff patch, extracts touched files and symbols, then walks the call graph and import graph to score every reachable file by how likely it is to be affected. Returns a ranked list with scores.

parameters
{
  "patch": "--- a/src/engine.rs\n+++ b/src/engine.rs\n@@ ..."
}
stitch_context Search + auto-attach cross-file callee definitions

Runs a semantic search and then automatically fetches the definitions of every symbol called by the top results. Returns the primary hits plus stitched dependency context — giving the agent a complete, multi-file view without additional round trips.

parameters
{
  "query": "how is the index built",
  "limit": 5
}

Fetches a symbol's source and stores documentation in .codixing/symbol_docs.json. When ANTHROPIC_API_KEY or OLLAMA_HOST is set, full LLM-generated docs are produced; otherwise a structured stub is written. Subsequent calls return the cached doc. Pass "force": true to bypass the cache.

parameters

  "symbol": "Engine::init",

}
remember Store persistent project knowledge

Stores a key/value note in .codixing/memory.json. Notes survive engine restarts and MCP reconnects — useful for recording architectural decisions, module conventions, and context that should not be lost between sessions.

parameters

Returns all matching memory entries. Searches by keyword substring (matched against key + value) and/or filters by tags. Call with no arguments to list everything.

parameters

  "query": "jwt",           // optional substring filter
  "tags":  ["security"]    // optional tag filter (AND)
}
forget Delete an outdated memory entry

Removes the memory entry with the given key. Returns a confirmation or a "not found" message.

parameters

find_tests Discover test functions

Finds test functions by naming conventions (test_*, *_test, TestXxx) and test annotations (#[test], @Test, @pytest.mark.*). Works across all supported languages.

parameters
{
  "pattern": "auth",       // optional name/file substring

}
find_similar Semantic duplicate / clone detection

Finds code chunks semantically similar to a named symbol. Uses vector embeddings (cosine similarity) when available, falls back to BM25. Useful for spotting copy-paste debt or finding parallel implementations that should be unified.

parameters
{

}
get_complexity Cyclomatic complexity per symbol

Computes cyclomatic complexity (McCabe 1976) for every function/method in a file by counting decision points in the source text. Returns a risk-banded table (low / moderate / high / critical) sorted by complexity descending.

parameters

}

Given a git diff, returns: (1) changed files, (2) symbols whose definitions overlap the diff hunks, (3) impact prediction (which other files may need changes), and (4) cross-file context for the most-changed symbols. Call at the start of a code review.

parameters
{
  "patch": "--- a/src/engine.rs\n+++ b/src/engine.rs\n@@ ..."
}

Assembles index statistics, language breakdown, top files by PageRank, and a token-budgeted repository map, then writes the result to .codixing/ONBOARDING.md. Run once after indexing a new project to produce a living reference document.

parameters

LSP Server

The codixing-lsp binary implements the Language Server Protocol over stdin/stdout, making Codixing's symbol table and search available in any LSP-capable editor.

Capabilities

  • textDocument/hover — kind, signature, and definition path for the symbol under the cursor
  • textDocument/definition — go-to-definition across all indexed files
  • textDocument/references — return a bounded set of indexed usages for a symbol
  • workspace/symbol — global symbol search by query string (up to 200 results)
  • textDocument/documentSymbol — per-file symbol outline
  • textDocument/completion — symbol-name completions from the index
  • textDocument/signatureHelp — parameter info on function calls
  • textDocument/codeAction — quickfix to suppress complexity warnings
  • textDocument/inlayHint — cyclomatic complexity scores on function lines
  • textDocument/rename + textDocument/prepareRename — rename refactoring with conflict detection
  • textDocument/semanticTokens/full — AST-based semantic tokens for Rust, Python, TypeScript/JavaScript, and Go

Call Hierarchy

textDocument/prepareCallHierarchy, callHierarchy/incomingCalls, and callHierarchy/outgoingCalls provide cross-language call hierarchy — navigate callers and callees across Rust, Python, TypeScript, and other supported languages.

Full-text open/change/save buffers are capped at 8 MiB per document. Oversized replacements are rejected from the resident document store rather than consuming unbounded editor-server memory.

Configuration

Codixing stores its index in a .codixing/ directory at the repository root. There is no config file required — all defaults are sensible. Advanced tuning is done via IndexConfig in library usage.

IndexConfig fields

FieldDefaultDescription
root—Primary repository root path
extra_roots[]Additional roots for multi-repo indexing
languages[]Optional allow-list; empty auto-detects supported formats
exclude_patternsbuilt-in defaultsAdditional glob patterns excluded from indexing
chunkdefaultsMaximum chunk size and overlap behavior
embeddingdisabledModel, batching, and vector-index settings
graphdefaultsPageRank damping and iteration count

.codixing/ directory layout

directory
.codixing/
├── active-generation.json    # atomically selected readable generation
├── generations/
│   └── gen-<id>/
│       ├── tantivy/          # BM25 segments and stored chunk content
│       ├── vectors/          # vector index snapshots, when enabled
│       ├── graph/            # file and symbol dependency graphs
│       ├── symbols_v2.bin    # persisted symbol table
│       ├── chunk_meta.bin    # compact chunk metadata
│       ├── file_trigram.bin  # shared exact-search file candidates
│       ├── tree_hashes_v2.bin # authoritative sync baseline
│       ├── config.json       # persisted index configuration
│       └── meta.json         # counts, format version, and git commit
└── …                         # writer locks, journals, and session state

Rebuilds and incremental checkpoints publish a complete new generation, then switch active-generation.json last. Readers therefore keep using the previous valid generation if publication is interrupted. A full rebuild temporarily needs room for both the active and replacement generations; on a large repository, budget roughly one additional index footprint of free disk. Use codixing doctor . --json to inspect the active generation and recovery status, codixing repair . to restore recoverable metadata and safely reclaim unpublished vector crash debris older than 24 hours, or codixing init . for a clean rebuild.

Embeddings

BGE embeddings (opt-in)

Enable fastembed ONNX vectors with codixing init . --embed --model bge-small-en. The model is downloaded on first use and stored in the system model cache; ordinary init remains BM25-only.

Install ONNX Runtime, then set ORT_DYLIB_PATH to the exact absolute path of libonnxruntime.so, libonnxruntime.dylib, or onnxruntime.dll. A Python wheel does not configure that environment variable for the Codixing process. Run codixing doctor to verify the resolved path. See the ort runtime-loading guide for platform details.

Model2Vec (ONNX-free)

For semantic retrieval without installing ONNX Runtime, use the static Model2Vec backend. It downloads a compact model on first use and builds vectors with the same durable embedding workflow.

shell
codixing init . --embed --model model2vec

# Larger retrieval-optimised static model
codixing init . --embed --model model2vec-retrieval

Qwen3-Embedding-0.6B feature flag

1024-dimensional embeddings via ONNX Runtime. Requires building with --features qwen3. Suitable when the higher-dimensional Qwen3 representation is preferred.

shell
cargo build --release -p codixing --features qwen3
codixing init . --embed --model qwen3

Qdrant client experimental

The qdrant feature compiles the experimental client APIs, but the current CLI does not route indexing or search through them. Do not use the BM25-only command below as a Qdrant setup path; backend selection still requires library integration.

shell
cargo build --release -p codixing-core --features qdrant

Architecture

Codixing is a Rust workspace with five crates:

CrateRole
coreEngine library — all retrieval, indexing, graph, and embedding logic
cliBinary wrapping core with clap subcommands
serverAxum REST API server exposing core over HTTP
mcpJSON-RPC 2.0 stdin/stdout MCP server for Claude Code
lspLanguage Server Protocol integration for editors

Retrieval pipeline

data flow
Query
  │
  ├─► BM25Retriever (Tantivy)
  │     └─ CodeTokenizer → camelCase/snake_case split
  │     └─ Field boosts: signature ×3, entity_names ×2
  │
  ├─► VectorRetriever (usearch HNSW)
  │     └─ BGE-Base-EN-v1.5 (fastembed ONNX)
  │     └─ Contextual prefix embedding
  │
  ├─► RRF Fusion (asymmetric k weights)
  │
  ├─► Graph boost: score *= 1 + 0.3 * pagerank
  │
  ├─► MMR deduplication (thorough+)
  │
  └─► Band merge + token budget → formatted output

Key design constraints

  • tree_sitter::Parser is !Send — a fresh parser is created per call, never stored in structs.
  • petgraph NodeIndex is fragile — remove_node() does a swap-remove; always keep path_to_node: HashMap in sync.
  • bitcode serialisation — uses bitcode::serialize/deserialize (serde feature), not encode/decode.
