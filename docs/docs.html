Installation

Codixing is distributed as a Rust workspace. There are no runtime dependencies — the final binary is statically linked and under 50 MB.

Build from source

You need the Rust stable toolchain. Install it with rustup if you haven't already.

shell
# Clone the repository
git clone https://github.com/ferax564/codixing
cd codixing

# Build all binaries with release optimisations
cargo build --release --workspace

# Binaries are now at:
./target/release/codixing       # CLI
./target/release/codixing-mcp   # MCP server
./target/release/codixing-server # REST API server

Optional features

Feature flagWhat it enables
--features qdrantQdrant vector backend instead of the built-in usearch HNSW index
--features qwen3Qwen3-Embedding-0.6B via candle (1024-dim, GPU-optional)
ℹ The default build uses BGE-Base-EN-v1.5 (768-dim) via ONNX through fastembed. No GPU or external model server required.

Quick Start

Index a repository, search it, and wire it up to Claude Code in under a minute.

shell
# 1. Index your project (BM25 only — instant, no model download)
codixing init . --no-embeddings
✓ Indexed 770 files, 14,203 symbols in 0.87s

# 2. Search with natural language
codixing search "authentication handler"
► src/auth/handler.rs:42  [score: 0.94]
pub fn handle_auth_request(req: Request) -> Result<Token>

# 3. Or with full hybrid (downloads BGE model on first run)
codixing init .
codixing search "authentication handler" --strategy fast

Connect to Claude Code

shell
# Start MCP daemon (keeps engine warm in memory)
codixing-mcp --root /path/to/your/project --daemon

Then add to your ~/.claude.json:

~/.claude.json
{
  "mcpServers": {
    "codixing": {
      "command": "/path/to/codixing-mcp",
      "args": ["--root", "/path/to/your/project", "--daemon"]
    }
  }
}

CLI — init

Initialises a Codixing index inside the .codixing/ directory at the given path. Safe to re-run — it performs an incremental update on an existing index.

usage
codixing init <PATH> [OPTIONS]
FlagDefaultDescription
--no-embeddingsfalseSkip vector embedding — BM25 only. Much faster init, no model download.
--also <DIR>—Add an extra root directory to the same index (multi-repo mode).
--forcefalseWipe and rebuild the index from scratch.
examples
# Fast BM25-only index of current directory
codixing init . --no-embeddings

# Full hybrid index
codixing init /path/to/project

# Multi-repo: index two roots together
codixing init . --also ../shared-lib

CLI — search

Query the index with a natural language phrase or symbol name. Results are ranked by score, graph centrality, and deduped with MMR.

usage
codixing search <QUERY> [OPTIONS]
FlagDefaultDescription
--strategy <S>fastOne of: instant, fast, thorough, explore, deep
--limit <N>20Maximum number of results to return
--tokens <N>8000Token budget for output (cl100k)
--lang <L>—Filter results to a specific language (e.g. rust, python)
--rerankerfalseEnable cross-encoder reranking (implies deep strategy)
examples
# Natural language query, default strategy
codixing search "parse JWT token"

# Identifier query — auto-routes to BM25-dominant
codixing search "TokenVerifier"

# Thorough hybrid with custom token budget
codixing search "database connection pool" --strategy thorough --tokens 4000

# Cross-encoder reranking for highest precision
codixing search "error handling middleware" --reranker

CLI — symbols

Look up definitions (functions, structs, classes, methods) by name. Case-insensitive substring match across all indexed files.

usage
codixing symbols <NAME> [OPTIONS]
FlagDefaultDescription
--kind <K>—Filter by symbol kind: function, class, struct, method, interface
--limit <N>20Max results
examples
# Find all symbols named "authenticate"
codixing symbols authenticate

# Only structs matching "config"
codixing symbols config --kind struct

CLI — callers / callees

Traverse the import and call graph to find who depends on a file, or what a file depends on.

usage
codixing callers <FILE> [--depth <N>]
codixing callees <FILE> [--depth <N>]
FlagDefaultDescription
--depth <N>1Traversal depth. --depth 2 returns transitive neighbours.
examples
# Who imports src/auth/handler.rs?
codixing callers src/auth/handler.rs

# What does handler.rs transitively depend on?
codixing callees src/auth/handler.rs --depth 2
jwt.rs → crypto.rs → openssl.rs
handler.rs → db/user.rs → models.rs

CLI — usages

Find all files and locations where a symbol is referenced. Combines BM25 with graph-boosted scoring.

usage
codixing usages <SYMBOL> [--limit <N>]
example
codixing usages "HybridRetriever" --limit 10
► src/retriever/hybrid.rs:14   [def]
► src/engine.rs:88             [call]
► tests/retrieval_test.rs:22   [call]

CLI — sync & git-sync

sync

Hash-based incremental sync. Walks all files, hashes content (xxh3), and only reindexes files that have changed or been deleted since the last run.

usage
codixing sync [PATH]

git-sync

Git-aware incremental sync. Uses git diff --name-status between the last indexed commit and HEAD. Faster than sync on large repos since it only processes files reported by git.

usage
codixing git-sync [PATH]

GitSyncStats { changed: 3, deleted: 1, unchanged: 766 }
Sync complete in 0.12s
ℹ git-sync requires the directory to be a git repository. Falls back gracefully with an error if not.

Search Strategies

Every search command and MCP tool accepts a strategy parameter. Choose based on your latency and precision requirements.

instant
BM25 only. No vector scoring, no graph boost.
~2ms · zero model required
fast
BM25 + vector RRF fusion + graph PageRank boost.
~10ms · default
thorough
Same as fast but with MMR deduplication applied.
~15ms
explore
BM25 first-pass then graph neighbour expansion (RepoHyper-style).
~20ms · best for architecture queries
deep
thorough + BGE-Reranker-Base cross-encoder second pass.
~80–150ms · highest precision

Asymmetric routing

Codixing automatically detects whether a query looks like an identifier (e.g. TokenVerifier, b2Vec2) and adjusts the RRF weights accordingly:

  • Identifier queries → BM25-dominant (k=20/90) — exact token match matters more
  • Natural language queries → vector-dominant (k=90/20) — semantic similarity matters more

Hybrid Retrieval

BM25

Tantivy-powered full-text index with a custom CodeTokenizer that splits on camelCase, snake_case, and punctuation. Field boosts ensure definitions outrank mentions:

  • signature field: ×3.0 boost
  • entity_names field: ×2.0 boost
  • content field: ×1.0

Vector

BGE-Base-EN-v1.5 (768-dim) embeddings via fastembed ONNX, stored in a usearch HNSW index with int8 quantisation (8× memory reduction). Chunks are embedded with contextual prefixes for better recall:

contextual prefix format
"{file_path} | {scope_chain} | {signature}\n{content}"

Reciprocal Rank Fusion (RRF)

BM25 and vector results are merged with RRF. The formula assigns a score to each candidate based on its rank in each list:

score(d) = Σ  1 / (k + rank_i(d))

Band merging (LDAR)

Adjacent chunks from the same file within 3 lines of each other are merged before output. This reduces token count by 25–91% on high-hit queries without losing context.

MCP — Setup

The codixing-mcp binary implements JSON-RPC 2.0 over stdin/stdout and is compatible with Claude Code and any MCP-capable client.

Standard mode

A new engine instance is loaded per invocation. Suitable for occasional use.

shell
codixing-mcp --root /path/to/project

Daemon mode

The first invocation starts a background daemon that holds the engine in memory and serves subsequent calls over a Unix socket (.codixing/daemon.sock). This gives 4–5× faster throughput on cheap operations.

shell
codixing-mcp --root /path/to/project --daemon

Auto-init

If no index exists at --root, the MCP server automatically runs init --no-embeddings before serving the first request. No manual setup required when connecting from Claude Code.

VS Code / Cursor extension

Install the extension from editors/vscode/. It exposes 6 commands and auto-writes the MCP server entry to both ~/.claude.json and ~/.cursor/mcp.json.

MCP — Tools Reference

All tools are called via JSON-RPC 2.0 tools/call with a name and arguments object.

code_search Hybrid search across all strategies

The primary retrieval tool. Runs BM25 + vector fusion with optional graph boost and MMR deduplication.

parameters
{
  "query":    "authentication middleware",  // required
  "strategy": "fast",                     // instant|fast|thorough|explore|deep
  "limit":    20,
  "tokens":   8000
}
find_symbol Case-insensitive definition lookup

Searches the symbol table for definitions matching the given name. Returns file path, line number, kind, and signature.

parameters
{
  "name": "HybridRetriever",
  "kind": "struct"  // optional: function|class|struct|method|interface
}
read_symbol Full source of any symbol by name

Returns the complete source block for a symbol (function body, struct definition, class, etc.), token-budgeted.

parameters
{
  "name":   "handle_auth_request",
  "tokens": 4000
}
get_repo_map Token-budgeted PageRank architecture map

Returns an Aider-style repository map: files sorted by PageRank with their top-level symbol signatures. Excellent for giving an agent a high-level orientation of a codebase.

parameters
{
  "tokens": 4000,
  "seed_files": ["src/engine.rs"]  // optional: personalised PageRank seed
}
get_references Callers + callees for any file

Returns the direct import/call graph neighbours of a file — who imports it, and what it imports.

parameters
{
  "file":  "src/auth/handler.rs",
  "depth": 1
}
get_transitive_deps Blast-radius multi-hop traversal

Returns all transitive dependencies (or dependents) of a file up to the specified depth. Useful for impact analysis before a refactor.

parameters
{
  "file":      "src/index/bm25.rs",
  "direction": "callers",  // "callers" | "callees"
  "depth":     3
}
search_usages All usage locations for a symbol

Combines BM25 with graph boost to find every location a symbol is referenced across the codebase.

parameters
{
  "symbol": "PageRank",
  "limit":  30
}
grep_code Regex/literal search with context lines

Token-budgeted grep with match highlighting, context lines, and glob filtering. Designed for LLM consumption — far more efficient than raw rg output.

parameters
{
  "pattern":  "fn handle_\\w+",
  "glob":     "**/*.rs",     // optional file filter
  "context":  2,              // lines before/after match
  "literal":  false,          // true = treat pattern as literal string
  "tokens":   4000
}
read_file Raw source with optional line range

Returns the raw source of a file, optionally sliced to a line range. Respects the token budget.

parameters
{
  "path":       "src/engine.rs",
  "start_line": 80,   // optional
  "end_line":   140,  // optional
  "tokens":     4000
}
index_status Diagnostic report with full index stats

Returns a snapshot of the current index state: file count, symbol count, vector count, graph node/edge counts, last sync commit, and strategy availability.

parameters
{}  // no parameters required

Configuration

Codixing stores its index in a .codixing/ directory at the repository root. There is no config file required — all defaults are sensible. Advanced tuning is done via IndexConfig in library usage.

IndexConfig fields

FieldDefaultDescription
root—Primary repository root path
extra_roots[]Additional roots for multi-repo indexing
chunk_config—Max chunk tokens, overlap lines
embedding_config—Model selection, batch size
graph_config—PageRank damping, max iterations

.codixing/ directory layout

directory
.codixing/
├── index/          # Tantivy BM25 index segments
├── vectors/        # usearch HNSW index + id map
├── graph/
│   └── graph.bin   # petgraph bitcode serialisation
├── symbols.bin     # DashMap symbol table (bitcode)
└── meta.bin        # IndexMeta: file hashes, last git commit

Embeddings

BGE-Base-EN-v1.5 (default)

768-dimensional embeddings via fastembed ONNX. Downloaded automatically on first init (no --no-embeddings flag). Stored in the system model cache. int8 quantised at index time.

Qwen3-Embedding-0.6B feature flag

1024-dimensional embeddings via candle. Requires building with --features qwen3. Suitable for GPU-accelerated environments or when higher-dimensional representations are preferred.

shell
cargo build --release --features qwen3
codixing init . --embedding-model qwen3

Qdrant backend feature flag

Replaces the built-in usearch HNSW index with an external Qdrant instance. Useful for team deployments where the vector index is shared.

shell
cargo build --release --features qdrant
QDRANT_URL=http://localhost:6333 QDRANT_COLLECTION=codixing codixing init .

Language Support

Codixing uses tree-sitter for AST-aware parsing. All 14 languages below get full chunking, symbol extraction, and import graph edges.

Tier 1 — Full support

Rust Python TypeScript TSX JavaScript Go Java C C++ C#

Tier 2 — Full support

Ruby Swift Kotlin Scala

Files in unsupported languages are indexed with a line-based fallback chunker — they appear in BM25 results but not in the symbol table or graph.

ℹ Zig and PHP are planned for the next roadmap phase. The .gitignore-aware walker (via the ignore crate, same as ripgrep) automatically skips build artefacts, vendor directories, and binary files.

Architecture

Codixing is a Rust workspace with four crates:

CrateRole
coreEngine library — all retrieval, indexing, graph, and embedding logic
cliBinary wrapping core with clap subcommands
serverAxum REST API server exposing core over HTTP
mcpJSON-RPC 2.0 stdin/stdout MCP server for Claude Code

Retrieval pipeline

data flow
Query
  │
  ├─► BM25Retriever (Tantivy)
  │     └─ CodeTokenizer → camelCase/snake_case split
  │     └─ Field boosts: signature ×3, entity_names ×2
  │
  ├─► VectorRetriever (usearch HNSW)
  │     └─ BGE-Base-EN-v1.5 (fastembed ONNX)
  │     └─ Contextual prefix embedding
  │
  ├─► RRF Fusion (asymmetric k weights)
  │
  ├─► Graph boost: score *= 1 + 0.3 * pagerank
  │
  ├─► MMR deduplication (thorough+)
  │
  └─► Band merge + token budget → formatted output

Key design constraints

  • tree_sitter::Parser is !Send — a fresh parser is created per call, never stored in structs.
  • petgraph NodeIndex is fragile — remove_node() does a swap-remove; always keep path_to_node: HashMap in sync.
  • bitcode serialisation — uses bitcode::serialize/deserialize (serde feature), not encode/decode.
