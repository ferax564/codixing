v0.22: fixing the OOM, the cold start,
and the accuracy gap

v0.21 shipped parallel embedding workers. They OOM'd on any repo over 30K chunks. The trigram index took 55 seconds to deserialize. Cross-package queries had 0.033 recall. v0.22 fixes all three: mpsc streaming, mmap trigram, and a new cross-imports command. Overall R@10 jumps from 0.456 to 0.607.

1. Parallel drain OOM → mpsc streaming

The v0.21 parallel drain was a classic collect-then-insert pattern. Each worker embedded its file batch, collected all (chunk_id, Vec<f32>, path) tuples into a thread-local Vec, then all Vecs merged into one Vec, then bulk-inserted into VectorIndex.

The math doesn't work. 881K chunks × 384 dims × 4 bytes = 1.35 GB just for the embedding vectors. Add the chunk IDs and paths, and you're well over the memory budget. OpenClaw (45K chunks) would OOM on 8 GB machines.

The fix: replace the collect-and-merge pattern with mpsc::sync_channel and back-pressure. Workers send per-file batches to an inserter thread that calls add_mut() immediately. No accumulation.

// mpsc streaming: workers send batches, inserter thread consumes immediately
let (tx, rx) = std::sync::mpsc::sync_channel::<EmbedBatch>(num_workers * 2);

// Inserter thread: drains channel, writes to VectorIndex as batches arrive
let inserter = std::thread::spawn(move || {
    let mut total = 0;
    while let Ok(batch) = rx.recv() {
        for (chunk_id, embedding, path) in batch.items {
            vec_idx.add_mut(chunk_id, &embedding, &path)?;
            total += 1;
        }
    }
    Ok(total)
});

// Workers: embed per-file, send batch, block if channel is full (back-pressure)
std::thread::scope(|s| {
    for _ in 0..num_workers {
        let tx = tx.clone();
        s.spawn(move || {
            let embedder = Embedder::new(model)?;
            loop {
                let jobs = pull_jobs(rq, 10)?;
                if jobs.is_empty() { break; }
                for job in jobs {
                    let batch = embed_file(&embedder, &job)?;
                    tx.send(batch)?;  // blocks if inserter is behind
                }
            }
        });
    }
});

The sync_channel capacity is num_workers * 2. If the inserter falls behind, workers block on send()—natural back-pressure, no unbounded growth. Peak memory: roughly num_workers × 2 batches in flight at once, about 50 MB instead of 1.35 GB.

Result: OpenClaw (45K chunks) now completes instead of OOM'ing.

2. Mmap trigram index — zero-cost cold start

The trigram index was serialized with bitcode and deserialized into a HashMap<[u8; 3], Vec<u64>>. For the Linux kernel (175 MB chunk_trigram.bin), that meant: read 175 MB from disk, decode bitcode, rebuild HashMap with 3M+ entries. 55 seconds.

v0.22 uses a custom mmap binary format that requires zero deserialization:

// Mmap trigram format (magic: "TRGM")
//
// Header: 20 bytes
//   bytes  0..4   magic "TRGM"
//   bytes  4..8   version (u32 LE)
//   bytes  8..12  num_trigrams (u32 LE)
//   bytes 12..16  num_postings_total (u32 LE)
//   bytes 16..20  reserved (u32 LE)
//
// Trigram index: num_trigrams entries, 12 bytes each, sorted by trigram
//   bytes 0..3   trigram [u8; 3]
//   byte  3       padding
//   bytes 4..8    offset into posting lists (u32 LE)
//   bytes 8..12   length of posting list (u32 LE)
//
// Posting lists: contiguous u64 LE values
//   each posting list is offset..offset+length chunk IDs

Lookup is O(log N) binary search over the sorted trigram entries. The file is mmap'd directly—the OS pages in only the regions that are accessed. Auto-detected by the "TRGM" magic bytes at file start; falls back to bitcode for old indexes. Backward-compatible.

For mutations (sync path), ensure_mutable() materializes the mmap data into in-memory HashMaps on first write. For the common search-only path, this is a no-op.

Result: 55s → 0.1s cold start (550× improvement).

3. cross-imports command — fixing the accuracy gap

This was the biggest regression in v0.21 benchmarks. Queries like "which gateway files import from security?" can't be answered by text search (grep sees import statements but doesn't understand direction) or per-file callers (too narrow). Cross-package R@10 was 0.033.

The problem is structural. Neither BM25 nor embeddings understand "files in package A that depend on package B." You need the import graph, and you need to query it directionally.

The solution is a dedicated command:

codixing cross-imports --from src/gateway --to src/security
  src/gateway/auth-middleware.ts    imports 3 symbols from src/security/
  src/gateway/rate-limiter.ts       imports 1 symbol  from src/security/
  src/gateway/request-validator.ts  imports 2 symbols from src/security/

Implementation: a single O(E) pass over the import graph edges. For each edge, check if the source file is under --from and the target file is under --to. No BM25, no embeddings, no ranking. Just graph traversal.

Result: cross-package R@10 went from 0.033 → 0.640. Overall R@10 jumped from 0.456 → 0.607.

4. populate_symbol_graph — eliminate double file reads

Graph construction was reading every file from disk twice. Phase 1 (definitions): read file, parse AST, extract symbols. Phase 2 (references): read file again, parse AST again, resolve references. On the Linux kernel (73K files), that's ~146K unnecessary disk reads.

The fix: pass the cached file_contents: &DashMap<PathBuf, String> from the indexing pipeline into populate_symbol_graph(). Both phases read from the cache. The DashMap was already populated during chunking—this is purely plumbing, no new allocations.

What's next

TypeScript import resolver. Three ground truth files in the benchmark still fail because our import resolver doesn't handle .js → .ts extension resolution (a TypeScript convention where imports use .js extensions but resolve to .ts files at build time). Fixing this should push cross-package R@10 above 0.8.

Embedding speed benchmarks. Now that the OOM is fixed and large repos actually complete, we can run proper wall-clock benchmarks for the embedding pipeline: time to embed OpenClaw (45K chunks) and the Linux kernel (881K chunks) with 1, 2, and 4 workers. We expect near-linear scaling up to 4 workers on machines with sufficient memory.
