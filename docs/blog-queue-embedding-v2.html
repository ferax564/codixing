How we replaced spawn_blocking
with RustQueue

Embedding 881K chunks on the Linux kernel takes 60+ minutes. If ONNX crashes at chunk 500K, you start over. We shipped a queue-based embedding pipeline, iterated through 6 versions to get it right, and ended up with 17× faster search and 67% better symbol recall than grep on a 368K LoC TypeScript monorepo.

The problem: synchronous embedding doesn't scale

Codixing uses BGE-Small-En embeddings for hybrid search—BM25 handles keyword retrieval, vector similarity handles natural language queries. For a small repo, embedding is fine: 1,054 chunks for our own codebase takes a few seconds.

For the Linux kernel (73K files, 881K chunks), it takes over 60 minutes. And our embedding path was entirely synchronous: one chunk at a time, one ONNX session, single-threaded. If the process died at chunk 500,000, you'd restart from zero.

We needed three things: crash recovery (resume from where we left off), parallelism (multiple ONNX workers), and a smart threshold so small repos don't pay the queue overhead.

v1: infrastructure without improvement

The first version integrated RustQueue—a Rust job queue backed by redb—and pushed one job per chunk. It worked. It was also slower than the direct path for anything under 10K chunks.

v1 was net-negative. Serializing 45K individual jobs to redb, pulling them one at a time, deserializing each—the overhead dwarfed the actual embedding time. We also eagerly loaded trigram indexes during init, adding 55 seconds of startup for data that wasn't needed until the first search.

Honest accounting: v1 shipped, was measured, and was worse. The queue infrastructure was sound but the job granularity and eager loading were wrong.

What we actually fixed in v2

1. File-grouped jobs with late chunking

Instead of one job per chunk, v2 pushes one job per source file. The worker reads the entire file, chunks it locally, and embeds all chunks together. This batches ONNX inference per file (better cache locality) and reduces queue I/O by 10-50×.

// push_file_embed_jobs: group pending chunks by file path
pub async fn push_file_embed_jobs(
    rq: &Arc<RustQueue>,
    pending: &DashMap<u64, String>,
    chunk_meta: &DashMap<u64, ChunkMeta>,
) -> Result<usize> {
    let mut file_chunks: HashMap<String, Vec<u64>> = HashMap::new();
    for entry in pending.iter() {
        if let Some(meta) = chunk_meta.get(entry.key()) {
            file_chunks
                .entry(meta.file_path.clone())
                .or_default()
                .push(*entry.key());
        }
    }

    let mut job_count = 0;
    for (file_path, chunk_ids) in &file_chunks {
        rq.push("embeddings", "late-chunk-file",
            json!({ "file": file_path, "ids": chunk_ids }),
            None,
        ).await.map_err(queue_err)?;
        job_count += 1;
    }
    Ok(job_count)
}

2. Parallel ONNX workers

Each worker gets its own Embedder (separate ONNX session, ~200 MB RAM each). Workers pull jobs from the queue concurrently using std::thread::scope, collect embeddings in thread-local buffers, then bulk-insert into the VectorIndex after all workers finish.

// drain_embed_queue_parallel: N workers, each with its own ONNX session
pub fn drain_embed_queue_parallel(
    rq: &Arc<RustQueue>,
    model: &EmbeddingModel,
    chunk_meta: &DashMap<u64, ChunkMeta>,
    vec_idx: &mut VectorIndex,
    contextual: bool,
    root: &Path,
    num_workers: usize,
) -> Result<usize> {
    // Phase 1: N workers embed in parallel
    let collected = std::thread::scope(|s| {
        let handles: Vec<_> = (0..num_workers)
            .map(|worker_id| {
                s.spawn(move || {
                    let embedder = Embedder::new(model)?;
                    let mut local = Vec::new();
                    loop {
                        let jobs = block_on_async(async {
                            rq.pull("embeddings", 10).await
                        })?;
                        if jobs.is_empty() { break; }
                        // embed each file, ack on success, fail on error
                        ...
                    }
                    Ok(local)
                })
            }).collect();
        // join all handles, collect results
        ...
    })?;

    // Phase 2: Sequential bulk insert into VectorIndex
    for (chunk_id, embedding, file_path) in &collected {
        vec_idx.add_mut(*chunk_id, embedding, file_path)?;
    }
    Ok(collected.len())
}

3. Smart threshold

The embed_pending function decides at runtime: if fewer than 1,000 chunks are pending, skip the queue entirely and embed synchronously. No redb I/O, no JSON serialization, no worker spawn overhead. This keeps small repos fast.

pub const QUEUE_THRESHOLD: usize = 1000;

pub fn embed_pending(rq: Option<&Arc<RustQueue>>, ...) -> Result<()> {
    if let Some(rq) = rq {
        if pending.len() >= QUEUE_THRESHOLD {
            let num_workers = available_parallelism()
                .map(|n| n.get().min(4))
                .unwrap_or(1);
            push_file_embed_jobs(rq, pending, chunk_meta).await?;
            drain_embed_queue_parallel(rq, model, ...)?;
            return Ok(());
        }
    }
    // direct sync path for small repos
    embed_and_index_chunks(pending, chunk_meta, embedder, vec_idx, ...)?
}

4. Deferred embeddings

BM25 init takes 27 seconds on OpenClaw. Embedding all 45K chunks takes another ~2 minutes. v2 defers embedding: BM25 search works immediately after init, and embeddings are computed in the background. Time to first search: 29 seconds (BM25-only) vs 31 seconds (with deferred embedding queued).

5. Parallel graph + trigram build

We moved graph construction and trigram index building into a rayon::join call. Both read from shared DashMaps but write to independent outputs—perfect for fork-join parallelism. This shaved seconds off init on large repos where both structures are substantial.

// Build graph and trigram indexes in parallel
let (graph, (trigram_idx, ft_idx)) = rayon::join(
    || {
        // Graph construction
        build_graph(&files, &root, &config, &parser, &pending_imports)

    || {
        // Trigram + file-trigram indexes (lazy build_batch)

Linux kernel scale

The real test: 73K files, 881K chunks, 935K symbols. BM25 init takes 534 seconds. Cold start search: 1.57 seconds. Warm search: 0.79 seconds.

The queue-based pipeline handles this scale because:

  • File-grouped jobs reduce queue entries from 881K to ~73K (one per file)
  • 4 parallel workers (capped at available_parallelism().min(4)) saturate the ONNX runtime without OOM
  • Crash recovery via redb: if a worker dies mid-batch, unacknowledged jobs stay in the queue
  • The chunk_meta.bin compact format keeps the metadata at 134 MB (was 1.5 GB before)

The failures along the way

I want to be explicit about what didn't work, because the "shipped v2" narrative hides 6 iterations of getting it wrong.

Failure 1: eager trigram loading

v1 loaded the trigram index during init. For the Linux kernel, that's a 2.8 GB file. It added 55 seconds to startup even though trigrams aren't needed until the first search. v2 uses OnceLock lazy loading—the trigram index loads on first access, not at init.

Failure 2: parallel drain OOM

The first parallel implementation spawned available_parallelism() workers (16 on an M4 Max). Each ONNX session is ~200 MB. 16 workers = 3.2 GB just for embedding sessions, on top of the 1+ GB of chunk metadata. We OOM'd on 16 GB machines. The fix: cap at 4 workers.

Failure 3: one job per chunk

45K individual redb writes + JSON serializations + 45K individual pulls. The queue overhead was larger than the embedding time. File grouping reduced this to ~2K jobs for OpenClaw.

Failure 4: async drain from sync context

RustQueue is async. Our indexing pipeline is sync. The naive tokio::runtime::Runtime::new().block_on() worked until we were called from an MCP handler that already had a tokio runtime. Nested runtimes panic. The fix: block_on_async that detects whether a runtime is active and uses block_in_place when it is.

pub(super) fn block_on_async<F, T>(fut: F) -> T
where F: Future<Output = T> {
    match tokio::runtime::Handle::try_current() {
        Ok(handle) => tokio::task::block_in_place(|| handle.block_on(fut)),
        Err(_) => {
            let rt = Runtime::new().expect("failed to create tokio runtime");
            rt.block_on(fut)
        }
    }
}

MRR is the sleeper stat: 0.496 vs 0.271. Mean Reciprocal Rank measures how high the first correct result appears. Codixing puts the right file at position 2 on average; grep puts it at position 4. When an AI agent consumes these results, that difference compounds across every tool call.

What's next

Cross-package retrieval. Our explore strategy fails on cross-package queries in monorepos (R@10 = 0.067). The graph currently doesn't traverse pnpm workspace boundaries. We need to build cross-package edges from import resolution.

Background drain. Right now, embed_pending blocks until all workers finish. For large repos, we want to return after BM25 init and drain the embedding queue in a background thread, upgrading search from BM25-only to hybrid as chunks get embedded.

Distributed workers. The current architecture caps at 4 workers on one machine. For enterprise-scale repos (millions of files), we want workers on separate machines pulling from the same queue. RustQueue's redb backend would need to be swapped for a network-accessible store, but the worker protocol is already designed for this.
