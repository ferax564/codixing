Your MCP tools aren't loading—
and your agent doesn't tell you

We spent an hour diagnosing why Claude Code kept using grep instead of Codixing's 54 MCP tools. The tools were configured. The binary worked. But Claude never saw them. The fix was two flags.

The silent failure

Here's the scenario: you install an MCP server, add it to .mcp.json, restart Claude Code, and assume it's working. You even wrote MANDATORY: Always use Codixing MCP tools in your CLAUDE.md. But Claude keeps using Grep, Glob, and Read.

What you don't see: Claude Code tried to connect to the MCP server at session start, the connection failed silently, and the server was dropped. Claude doesn't announce "hey, your MCP server didn't load"—it just proceeds without it.

We caught this by searching for our MCP tools in Claude's deferred tool list:

# Expected: mcp__codixing__code_search, mcp__codixing__find_symbol, ...
ToolSearch("codixing")
→ No matching deferred tools found

Zero. Not a single Codixing tool loaded. Meanwhile, other MCP servers (Context7, Playwright) loaded fine.

Root cause 1: Stale daemon socket

The Codixing MCP server has a daemon mode that keeps the index in memory for fast subsequent calls (~1ms vs ~30ms cold start). When the daemon exits, it should clean up its socket file. But if the process crashes or gets killed, a stale daemon.sock stays behind.

What happens at session start: 1. Claude Code spawns: codixing-mcp --root . --compact 2. Binary sees daemon.sock exists → tries to proxy through it 3. Connection refused (no daemon process running) 4. Binary attempts to auto-fork a new daemon... 5. Waits up to 1s for daemon to start (20 × 50ms) 6. Claude Code's MCP client times out first → drops the server 7. Daemon eventually starts, but nobody's listening on stdin 8. Result: 0 tools available for the entire session

Fix: --no-daemon-fork. MCP clients like Claude Code manage their own process lifecycle—they don't need the server to auto-fork daemons. Run in direct stdio mode and let the client handle restarts.

Root cause 2: The discovery tax

Even when the server loads successfully, --compact mode creates friction. It exposes only 2 meta-tools: search_tools and get_tool_schema. To use any real tool, Claude must:

--compact: 3 round-trips per task Step 1: search_tools("search") → discover code_search exists Step 2: get_tool_schema(["code_search"]) → get the parameter schema Step 3: code_search("Engine struct") → actually do the work --medium: 1 round-trip per task Step 1: code_search("Engine struct") → schema already loaded, just call it

Meanwhile, Grep, Glob, and Read are available immediately with full schemas. LLMs take the path of least resistance. If your tool requires 3 round-trips to use, it won't be used.

Mode Tokens at startup Tools visible Round-trips per task
--compact ~200 2 (meta only) 3
--medium ~2,600 17 (core tools) 1

The ~2,400 extra tokens at startup pay for themselves after a single task.

The fix: two flags

// .mcp.json (or plugin.json)
{
  "mcpServers": {
    "codixing": {
      "command": "npx",
      "args": ["-y", "codixing-mcp", "--root", ".",
               "--medium", "--no-daemon-fork"]
    }
  }
}

That's it. --medium exposes 17 core tools with schemas. --no-daemon-fork runs in direct stdio mode. All 54 tools remain callable—the other 37 are accessible via search_tools + get_tool_schema on demand.

The adoption problem was infrastructure, not prompting. Writing "MANDATORY: use Codixing tools" in CLAUDE.md does nothing if the tools aren't loaded. Fix the connection first, then the instructions take effect.

Lessons for MCP server authors

1. Don't auto-fork daemons in MCP stdio mode

MCP clients manage process lifecycle. Your server should run in direct mode when invoked as a subprocess. Save daemon mode for CLI usage where users call the tool repeatedly.

2. Expose enough tools to be competitive

If your MCP server only shows 2 meta-tools, LLMs will ignore them and use built-in tools instead. Expose your 15-20 most important tools directly. The token cost is negligible compared to the adoption gain.

3. Failures are silent

Claude Code doesn't announce "your MCP server failed to load." Neither does Cursor. Neither does any client we've tested. If your server doesn't load, the agent proceeds without it and nobody notices—unless they check the tool list explicitly.

4. Clean up after yourself

If your server uses Unix sockets, named pipes, or lock files, handle stale artifacts. A socket file without a listener is worse than no socket at all—it blocks new connections with "Connection refused" instead of letting the server start fresh.

Upgrading

Update to v0.22.1 to get the new defaults:

# Plugin users (recommended)
claude plugin install codixing@codixing

# MCP-only users
claude mcp add codixing -- npx -y codixing-mcp --root . --medium --no-daemon-fork

If you're already on v0.22.0, just add --medium --no-daemon-fork to your .mcp.json args and restart Claude Code.
